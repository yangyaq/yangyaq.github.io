tornado官网阅读笔记

2019-08-20

前言

线程和WSGI

  1. Tornado框架不同于其它框架： 它不基于WSGI，并且它是典型的每个进程中单线程的运行方式。
  2. Tornado 代码不是线程安全的，一个线程安全地调用另外一个线程的 方法可通过IOLoop.add_callback，也可以用IOLoop.run_in_executor，但是Tornado 代码不是线程安全的，使用此方法时应避免引用Tornado中的任何对象。

整合asyncio

  1. Tornado整合了标准库模块asyncio 并 使用了相同的事件循环（>=5.0版本）

  2. Tornado 应该运行在所有的类UNIX操作系统，生产环境推荐Linux（使用epoll）和 BSD（kqueue）, 在Linux中有最好的性能和可扩展性; Mac OS X 支持 kqueue ，但网络性能仅适用于开发模式 ; Tornado也可以运行在Windows（也是推荐开发时使用）

异步和非阻塞IO

即时网络需要为每个用户建立一个连接，而这个连接大多数时间是空闲的，传统web服务器通过为每个用户提供一个线程来实现，这样看来，耗费的资源无疑是巨大的。

为了降低并发连接所耗费的资源，Tornado使用了单线程的事件循环机制, 也代表着代码要写成异步非阻塞的，因为单线程一次只能激活一个操作。

注意，异步和非阻塞有着紧密的联系，但并不相同。

阻塞

同步函数等待阻塞完成之后才返回，造成阻塞的原因也比较多，但通常都是在函数运行时和占用CPU时会发生阻塞，也有网络IO阻塞，磁盘IO阻塞，互斥等等。

同一个函数可能会发生网络阻塞，而不会发生磁盘IO阻塞，也就是说，一个函数在某些方面阻塞，在某些方面不阻塞。虽然所有的阻塞都应最小化，但在Tornado中主要讨论的是网络IO阻塞。

异步

异步函数在完成之前返回， 并触发一些动作, 这也是协程异步的原因，在这里有很多风格的异步接口。

  1. Callback 参数
  2. 返回占位符（Future, Promise, Deferred）
  3. 传递到queue
  4. 注册回调 （例如 POSIX signal）

像gevent这样的使用系统轻量级线程来提供与异步系统相当的性能，但实际上并没有异步化。gevent 可以不用修改代码直接打补丁来改变行为，但Tornado需要改变编写代码的方式，才能异步化。

Tornado中的异步操作返回占位符对象（Future），除了一些底层组件（例如IOLoop）, Future 使用 关键字await和yield 来转换结果。

HTTP缓存

2019-08-20

HTTP缓存

用途

减少一些静态文件向服务器请求的次数，减少服务器压力。

原理

客户端向服务端请求获取某f文件，服务器把f文件返回给浏览器，浏览器把请求后拿到的f文件存储到本地，下次请求时看看本地是否有缓存文件，如果有，就直接拿本地的文件。

两种缓存方式

强制缓存（Expires）和协商缓存（Cache-Control），强制缓存分为

强制缓存

Expires缓存

它的值是一个绝对时间的GMT格式的时间字符串，如Expires: Mon, 06 Feb 2017 08:26:48 GMT。这个时间代表资源的失效时间。在此之间，即命中强缓存。（缺点：当客户端与服务器时间出现较大偏差，就会出现混乱。）

cache-control缓存原理（时间维度）

解决了Expire的缺点，采用相对时间（max-age）

设置一个缓存时间段，到期后，浏览器发起获取文件的请求。

字段cache-control，可选值如下:

  1. public: 所有内容都将被缓存（客户端和代理服务器都可以缓存）
  2. private: 内容只缓存到私有缓存中（仅客户端可以缓存，代理服务器不可缓存）
  3. no-cache： 使用对比缓存，如果有验证令牌（Etag），no-cache会发起请求验证缓存是否有效，若有效（也就是资源未更改），可避免下载
  4. no-store: 所有内容都不会被缓存到缓存或者internet临时文件中
  5. max-age=xxx：缓存的内容将在 xxx 秒后失效, 这个选项只在HTTP 1.1可用, 并如果和Last-Modified一起使用时, 优先级较高

第一次访问请求，客户端向服务端发起一个文件请求，服务器返回文件并在response中添加响应头Cache-Contorol:max-age=60，然后这个文件在浏览器缓存60s，60s之内的请求文件请求将在缓存中直接拿，60s后客户端需要重新向服务器发起请求。

特殊点：

1. 但当`Cache-Control:max-age=0`或者`Cache-Control:no-store`时，不论服务器返回的响应中`max-age`设置多少都不会缓存。
2. 浏览器对在地址栏直接输入的文件地址请求做了优化处理，自动加上了`Cache-contorl:max-age=0`，就是不会缓存，直接获取最新的。

协商缓存

  • 通过看文件修改时间（有时文件修改了，又修改回去了，修改时间发生了变化，实际上文件内容没有发生变化）

    浏览器向服务器请求获得文件后，服务器返回该文件的最后修改时间Last-Modified，作为文件的一个标识 ，下次浏览器请求的时候 ，带着标识去请求（此时为If-Modiftied-Since）,然后服务器做校验。

    1. 如果说时间标识If-Modiftied-Since等于服务器端的文件修改时间，则文件没有发生改变，返回304，浏览器从缓存中取新的文件；
    2. 如果说时间标识If-Modiftied-Since小于服务器端的文件修改时间，那么文件发生了修改，浏览器会重新获取新的文件。
    3. 如果说时间标识If-Modiftied-Since大于服务器端的文件修改时间，认为请求错误。

  • 通过查看文件内容和文件修改时间是否修改

    浏览器向服务器请求获得文件后，服务器返回该文件的最后修改时间Last-Modified，作为文件的一个标识 ，返回etag作为文件内容的md5值；下次浏览器请求的时候 ，带着标识去请求（此时为If-Modiftied-Since）,再带着If-None-Match:etag值然后服务器做校验。

    流程如下：

    1. 如果说时间标识If-Modiftied-Since等于服务器端的文件修改时间，则文件没有发生改变，返回304，浏览器从缓存中取上次缓存的文件；
    2. 如果说时间标识If-Modiftied-Since小于服务器端的文件修改时间，此时再对比If-None-Match，若发生了变化，那么文件发生了修改；浏览器会重新获取新的文件。
    3. 如果说时间标识If-Modiftied-Since大于服务器端的文件修改时间，认为请求错误。

缓存优先级

Cache-Control是关于浏览器缓存的最重要的设置，因为它覆盖其他设置，比如 Expires 和 Last-Modified。另外，由于浏览器的行为基本相同，这个属性是处理跨浏览器缓存问题的最有效的方法。

DNS解析

输入url

DNS预取技术： 利用现有的DNS机制，提前解析网页中的可能的网络连接。

​ 当在浏览器中输入网址时，浏览器从历史记录，书签中查找记录，然后给出智能提示，智能补全地址url地址，用户还没有按下 enter 键， 浏览器已经开始使用 DNS 预取技术解析该域名了。

查询DNS对应请求的IP地址

迭代查询/递归查询

  • 先搜索自己的DNS缓存
  • 不匹配就去本机的hosts有没有对应的域名映射
  • 没有，去路由器查找路由器的DNS缓存
  • 没有，操作系统将域名发送给至本地域名服务器—递归方式查询本地域名服务器的查找自己的DNS缓存，查找成功则返回结果，否则， 采用迭代查询方式。
  • 本地域名服务器将得到的IP地址返回给操作系统并缓存起来
  • 操作系统再给浏览器并缓存起来
  • 浏览器得到了IP地址

建立TCP连接

TCP是可靠的，传输层协议。

三次握手

TCP报文

  • 建立一个TCP连接时，需要客户端和服务器端总共发送3个包。
    三次握手的目的是连接服务器指定端口，建立TCP连接，并同步连接双方的序列号和确认号，交换TCP窗口大小信息。在socket编程中，客户端执行connect()时将触发三次握手。
  • 第一次握手（SYN=1，seq=x）：
    客户端发送一个TCP的SYN标志位置1的包，指明客户端打算连接的服务器的端口，以及初始序号X，保存在包头的序列号（Sequence Number）字段里。
  • 第二次握手（SYN=1，ACK=1，seq=y，ACKnum=x+1）：
    服务器发回确认包（ACK）应答。即SYN标志位和ACK标志位均为1。服务器端选择自己的ISN序列号，放在seq域里，同时将确认序号（Acknowledgement Number）设置为客户的ISN加1，即X+1。发送完毕后，服务器端进入SYN_RCVD状态。
  • 第三次握手（ACK=1，ACKnum=y+1）：
    客户端再次发送确认包（ACK），SYN标志位为0，ACK标志位为1，并且把服务器发来ACK的序号字段+1，放在确定字段中发送给对方，并且在数据段放写ISN的+1。
    发送完毕后，客户端进入ESTABLISHED状态，当服务器端收到这个包时，也进入ESTABLISHED状态，TCP握手结束，TCP连接建立完成。

四次挥手

释放一个TCP连接，需要客户端和服务器总共发送4个包。客户端和服务器端均可主动发起挥手动作。在socket编程中，任何一方执行close()操作即可产生挥手操作。

  • 第一次挥手（FIN=1，seq=x）：
    假设客户端想要关闭连接，客户端发送一个FIN标志位置为1的包，表示自己已经没有数据可以发送了，但是仍然可以接收数据。
    发送完毕之后，客户端进入FIN_WAIT_1状态。
  • 第二次挥手（ACK=1，ACKnum=x+1）：
    服务器端确认客户端的FIN包，发送一个确认包，表明自己接收到了客户端关闭连接的请求，但还没有准备好关闭连接。
    发送完毕后，服务器端进入CLOSE_WAIT状态，客户端接收到这个确认包之后进入FIN_WAIT_2状态，等待服务器端关闭连接。
  • 第三次挥手（FIN=1，seq=y）：
    服务器端准备好关闭连接时，向客户端发送结束连接请求，FIN置为1。
    发送完毕后，服务器端进入LAST_ACK状态，等待来自客户端的最后一个ACK。
  • 第四次挥手（ACK=1，ACKnum=y+1）：
    客户端接收到来自服务器的端的关闭请求，发送一个确认包，并进入TIME_WAIT状态，等待可能出现的要重传的ACK包。
    服务器端接收到这个确认包之后，关闭连接，进入CLOSED状态。
    客户端等待了某个固定时间（两个最大段生命周期，2MSL，2Maximum Segment Lifetime）之后，没有收到服务器端的ACK，认为服务器端已经正常关闭连接，于是自己也关闭连接，进入CLOSED状态。

发送请求报文

收到响应报文，渲染页面

响应报文

​ 由三个部分组成，分别是：状态行、消息报头、响应正文。

​ 响应正文：就是服务器返回的资源的内容。

渲染

webkit浏览器渲染过程

  1. 解析HTML，构建 DOM 树
  2. 解析 CSS ，生成 CSS 规则树
  3. 合并 DOM 树和 CSS 规则，生成 render 树
  4. 布局 render 树（ Layout / reflow ），负责各元素尺寸、位置的计算
  5. 绘制 render 树（ paint ），绘制页面像素信息
  6. 浏览器会将各层的信息发送给 GPU，GPU 会将各层合成（ composite ），显示在屏幕上

构建DOM树

在 WebKit 中这一过程如下：首先是字节流，经过解码之后是字符流，然后通过词法分析器会被解释成词语（Tokens），之后经过语法分析器构建成节点，最后这些节点被组建成一棵 DOM 树。

浏览器在解析html过程中，会自上而下加载，并在加载的过程中进行解析渲染，如果遇到请求外部资源时（图片、外链的CSS、iconfont等）请求时异步的，并不会影响html文档进行加载，且统一交由Browser进程来处理，这样使得资源在不同网页间共享变得容易。

解析 CSS

从css字符串经过css解释器处理后变成渲染引擎内部规则的表示过程

RenderObject 树建立之前，CSS 解释器解释后的结果会保存起来，然后 RenderObject 树基于该结果来进行规范匹配和布局计算。

渲染过程中遇到javascript

当文档加载过程中遇到js文件，html文档会挂起正在渲染的线程，等待js文件加载、执行完毕、才可以恢复html文档的渲染过程。因为JS可能会修改DOM（document.write）这意味着，在 JS 执行完成前，后续所有资源的下载可能是没有必要的，这是 js 阻塞后续资源下载的根本原因。所以我们平时的代码中，js 是放在 html 文档末尾的。

渲染合成Render树

DOM树构建完成后，Webkit 会为DOM树节点构建RenderObject树，再通过RenderObject树构建出RenderLayer树

RenderObject 树是基于 DOM 树建立起来的一棵新树，是为了布局计算和渲染等机制而构建的一种新的内部表示。RenderObject 树节点和 DOM 节点不是一一对应关系，因为有可视节点（常用的 div img 标签等）与不可视节点（如 head、meta 标签），不可视节点是不会构成 RenderObject 树的。

布局

当 WebKit 创建 RenderObject 对象之后，每个对象是不知道自己的位置、大小等信息的，WebKit 根据框模型来计算它们的位置，大小等信息的过程称为布局计算。

布局计算是一个递归的过程，因为一个节点的大小通常需要先计算它的子女节点的位置，大小等信息。

当用户 网页的动画、翻滚网页、JavaScript 代码通过 CSSDOM 等操作时还会有重新布局。

绘图

在webkit中，绘图操作就绘图上下文

绘图上下文有两种类型：

  • 2D图形上下文，用来绘制2D图形的上下文
  • 3D绘图上下文，用来绘制3D图形的上下文

2D绘图上下文具体的作用：提供绘图单元的绘图接口以及设置绘图的样式，绘图接口包括画点，线，图片，多边形，文字等，绘图样式包括颜色、线宽、字号、渐变等

网页的渲染方式有3种

  • 软件渲染（由CPU完成）
  • 硬件加速渲染（由GPU来完成）
  • 混合模式

理想情况下，每个层都有个绘制的存储区域，这个存储区域用来保存绘图的结果。最后，需要将这些层的内容合并到同一个图像之中，可以称之为合成（Compositing），使用了合成技术的渲染称之为合成化渲染。

所以，在完成构建 DOM 树之后，WebKit 会调用绘图操作、软件渲染或者硬件加速渲染或者两者都有，将模型绘制出来，呈现在屏幕上。
至此，浏览器渲染完成。

Dijkstra实现

  1. 将源点加入堆，并调整堆。

  2. 选出堆顶元素u（即代价最小的元素），从堆中删除，并对堆进行调整。

  3. 处理与u相邻的，未被访问过的，满足三角不等式的顶点

    1):若该点在堆里，更新距离，并调整该元素在堆中的位置。

    2):若该点不在堆里，加入堆，更新堆。

  4. 若取到的u为终点，结束算法；否则重复步骤2、3。

示例

1: visited=[] heap=[(0,E)]

2: 弹出(0,E)，visited=[E] heap=[(1,B),(3,F)]

3: 弹出(1,B)，visited=[E,B] heap=[(3,A),(4,D),(3,F)]

4: 弹出(3,A)，visited=[E,B,A] heap=[(8,C),(4,D),(3,F)]

5: 弹出(3,F)，visited=[E,B,A,F] heap=[(6,C),(8,C),(4,D)]

6: 弹出(6,C)，visited=[E,B,A,F,C] , C为终点， 返会路径长度为6

e为边数

每次取出最短路径的复杂度降为O(1) , 堆调整的复杂度为O(elogn)

时间复杂度最大为O(nloge)

乐观锁与悲观锁

2019-08-20

悲观锁

​ 修改数据库数据之前对这条数据加锁，避免别人同时修改。

​ 这种利用数据库锁机制在修改数据之前先锁定，再修改的方式称为悲观并发控制，又名“悲观锁”，Pessimistic Concurrency Control，缩写“PCC”）。

​ 之所以叫做悲观锁，是因为这是一种对数据的修改抱有悲观态度的并发控制方式。我们一般认为数据被并发修改的概率比较大，所以需要在修改之前先加锁。

​ 悲观并发控制实际上是“先取锁再访问”的保守策略，为数据处理的安全提供了保证。

​ 线程1加排它锁->加锁成功后进行数据更新->更新事务提交，锁释放->线程2加锁成功->线程2更新

​ 效率方面，处理加锁的机制会让数据库产生额外的开销，还有增加产生死锁的机会；另外降低了并行性，一个事务锁定了某行数据，其他事务就必须等待该事务处理完才可以处理那行数据。

乐观锁

​ 乐观锁（ Optimistic Locking ） 是相对悲观锁而言的，乐观锁假设数据一般情况下不会造成冲突，所以在数据进行提交更新的时候，才会正式对数据的冲突与否进行检测，如果发现冲突了，则让返回用户错误的信息，让用户决定如何去做。

​ 相对于悲观锁，在对数据库进行处理的时候，乐观锁并不会使用数据库提供的锁机制。一般的实现乐观锁的方式就是记录数据版本。

​ ￼乐观并发控制相信事务之间的数据竞争(data race)的概率是比较小的，因此尽可能直接做下去，直到提交的时候才去锁定，所以不会产生任何锁和死锁。

悲观锁实现方式

悲观锁的实现，往往依靠数据库提供的锁机制。在数据库中，悲观锁的流程如下：

  • 在对记录进行修改前，先尝试为该记录加上排他锁（exclusive locking）。
  • 如果加锁失败，说明该记录正在被修改，那么当前查询可能要等待或者抛出异常。具体响应方式由开发者根据实际需要决定。
  • 如果成功加锁，那么就可以对记录做修改，事务完成后就会解锁了。
  • 其间如果有其他对该记录做修改或加排他锁的操作，都会等待我们解锁或直接抛出异常。

我们拿比较常用的MySql Innodb引擎举例，来说明一下在SQL中如何使用悲观锁。

要使用悲观锁，我们必须关闭mysql数据库的自动提交属性，因为MySQL默认使用autocommit模式，也就是说，当你执行一个更新操作后，MySQL会立刻将结果进行提交。set autocommit=0;

乐观锁实现方式

使用乐观锁就不需要借助数据库的锁机制了。

乐观锁的概念中其实已经阐述了他的具体实现细节：主要就是两个步骤：冲突检测和数据更新。其实现方式有一种比较典型的就是Compare and Swap(CAS)。

CAS是项乐观锁技术，当多个线程尝试使用CAS同时更新同一个变量时，只有其中一个线程能更新变量的值，而其它线程都失败，失败的线程并不会被挂起，而是被告知这次竞争中失败，并可以再次尝试。

比如前面的扣减库存问题，通过乐观锁可以实现如下：

以上更新语句存在一个比较重要的问题，即传说中的ABA问题。

比如说一个线程one从数据库中取出库存数3，这时候另一个线程two也从数据库中库存数3，并且two进行了一些操作变成了2，然后two又将库存数变成3，这时候线程one进行CAS操作发现数据库中仍然是3，然后one操作成功。尽管线程one的CAS操作成功，但是不代表这个过程就是没有问题的。

如何选择

在乐观锁与悲观锁的选择上面，主要看下两者的区别以及适用场景就可以了。

1、乐观锁并未真正加锁，效率高。一旦锁的粒度掌握不好，更新失败的概率就会比较高，容易发生业务失败。

2、悲观锁依赖数据库锁，效率低。更新失败的概率比较低。

随着互联网三高架构（高并发、高性能、高可用）的提出，悲观锁已经越来越少的被使用到生产环境中了，尤其是并发量比较大的业务场景。

  • GROUP BY子句中列出的每个列都必须是检索列或有效的表达式。(但不能是聚集函数)。如果在SELECT中使用表达式，则必须在GROUP BY子句中指定相同的表达式。不能使用别名。

  • 如果分组列中具有NULL值，则NULL将作为一个分组返回。如果列中有多行NULL值，它们将分为一组。
  • GROUP BY子句必须出现在WHERE子句之后，ORDER BY子句之前。

having过滤分组

WHERE与HAVING的区别

  • WHERE在数据分组前进行过滤，HAVING在数据分组后进行过滤
  • WHERE排除的行不包括在分组中。这可能会改变计算值，从而影响HAVING子句中基于这些值过滤掉的分组。

SELECT子句顺序

子句 说明 是否必须使用
select 要返回的列或者表达式 是
from 从中检索数据的表 仅在从表选择数据时使用
where 行级过滤 否
group by 分组说明 仅在按组计算聚集时使用
having 组级过滤 否
order by 输出排序顺序 否
limit 要检索的行数 否

组合查询

将结果组合在一个结果集中（可应用于不同表）。重复行会自动取消

tip：使用union all 时不取消重复行

union规则：

  • UNION必须由两条或两条以上的SELECT语句组成，语句之间用关键字UNION分隔(因此，如果组合4条SELECT语句，将要使用3个UNION关键字)。

  • UNION中的每个查询必须包含相同的列、表达式或聚集函数(不过各个列不需要以相同的次序列出)。

  • 列数据类型必须兼容:类型不必完全相同，但必须是DBMS可以

    隐含地转换的类型(例如，不同的数值类型或不同的日期类型)。

全文本搜索

需要建立索引，FULLTEXT(col)，仅在MyISAM引擎中才支持。

使用 where Match(col) Against(‘rabbit’) : 搜索包含词rabbit的行

扩展查询

where Match(col) Against(‘rabbit’ WITH QUERY EXPANSION)

布尔文本搜索

没有定义FULLTEXT索引也可以使用，但是非常缓慢。性能随着数据量的增加而降低。

使用：

where Match(col) Against(‘heavey -rope* IN BOOLEAN MODE’ )

布尔操作符 说明
+ 包含，词必须存在
- 不包含，词必须不出现
> 包含，而且增加等级值
< 包含，且减少等级值
() 把词组成子表达式(允许这些子表达式作为一个组被包含、排除、排列等)
～ 取消一个词的排序值
* 词尾的通配符
“” 定义一个短语(与单个词的列表不一样，它匹配整个短语以便包含或者排除这个短语

搜索包含词rabbit和bait的行

没有指定操作符，搜索匹配包含的rabbit和bait中的至少一个词的行

搜索匹配短语rabbit bait的行

匹配rabbit和carrot，增加前者的等级，降低后者的等级

匹配搜索词safe和combination，降低后者的等级

全文本搜索的使用说明

  • 索引全文本时短词（3个和小于3个字符的词，3这个数目可以设置）被忽略且从索引中排除。
  • MYSQL的内建的非用词列表（可以覆盖这个列表），在索引全文本数据时会被忽略。
  • 许多词出现的频率很高，搜索它们没有用处，因此，mysql规定了一条50%规则，如果一个词出现在50%以上的行中，则将它作为一个非用词省略，50%规则不用于IN BOOLEAN MODE。
  • 如果表中的行数少于3行，则全文本 搜索不返回结果（因为每个词出现或者不出现，或者至少出现在50%的行中）
  • 忽略单词中的单引号。例如（don‘t 索引尾dont）
  • 不具有词分隔符的语言（汉语，日语）不能恰当返回全文本搜索的结果
  • 仅在MyISAM数据库中支持全文本搜索。

  • 缺点：工厂类不够灵活，增加新的具体产品需要修改工厂类的判断逻辑代码，而且产品较多时，工厂方法代码将会非常复杂。

工厂方法模式

对简单工厂进行优化，取消统一的工厂类，针对不同的产品提供与之对应的工厂类。

定义：

​ 定义了一个创建对象的接口，但由子类决定将哪个类实例化。让一个类的实例化推迟到子类。

使用示例:

​ 仅仅需要知道工厂名，就可以创建出对应的实例。

四个角色：

  • 抽象产品类: DB
  • 具体产品类: MysqlDB MongoDB
  • 抽象工厂类 : AbstractDBFactory
  • 具体工厂类 : MongoFactory MysqlFactory

应用场景：

用来处理对象的创建，并将这样的行为封装在子类中。这样客户端程序中关于超类的代码就和子类对象的创建代码解藕了。

抽象工厂模式

定义：

​ 提供一个创建一系列相关或者相互依赖的接口，而无需指定他们具体的类。（在抽象工厂模式中，每一个具体工厂都提供了多个工厂方法用于产生多种不同类型的对象）

四个角色：

  • 抽象产品： 为每种对象声明接口，在其中声明了对象的所有业务方法

  • 具体产品： 它定义具体工厂生产的具体对象。

  • 抽象工厂：声明了一组用于创建对象的方法，注意是一组。

  • 具体工厂：它实现了在抽象工厂中声明的创建对象的方法，生成一组具体对象。

单例模式

定义:

​ 确保一个类只有一个实例，并提供一个全局访问点。

总结

抽象工厂模式和工厂方法模式区别

  • 抽象工厂模式：提供一个接口，用于创建相关或依赖对象的家族，而不需要明确指定具体类。
  • 工厂方法模式：定义了一个创建对象的接口，单由子类决定要实例化的类是哪一个。工厂方法让类吧实例化推迟到子类。

OO原则：

  • 多用组合，少用继承。
  • 针对接口编程，不针对实现编程。
  • 依赖抽象，不要依赖具体产品
